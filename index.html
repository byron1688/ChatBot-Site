1 System Architecture

Multi-Agent Routing & RAG on 32 GB

The core challenge: run multiple LLMs simultaneously for different tasks (routing, generation, memory extraction, TTS) within 32 GB of unified memory. Every model choice is a tradeoff between quality, speed, and memory footprint.

The Routing Pipeline

Each user message passes through a 3-layer classification cascade before hitting the generation model. The goal: route fast, cheap messages to tiny models and reserve the 8B for complex tasks.

1
Semantic Router (nomic-embed-text, 768d)
Embeds user input and computes cosine similarity against ~130 pre-embedded example utterances across 5 intent categories. Cache built on first startup, loads in ~300ms on CPU. Catches ~78% of intents with zero LLM calls.
2
LLM Classifier (Qwen 3.5 0.8B, ~50ms)
If semantic confidence is below threshold, a tiny 0.8B model classifies intent with temperature=0.0 and num_predict=10. Combined with keyword boosting to catch under-classified edge cases.
3
Keyword Fallback (zero-latency)
Regex + keyword matching as the ultimate safety net when Ollama is unavailable. Covers ~85% of common intents with hand-tuned bilingual (Chinese + English) keyword lists.
4
Turn Understanding (structured JSON extraction)
Extracts task_type, user_goal, entities, missing_info, needs_web, needs_stronger_model, avatar_action, and avatar_mood into a typed JSON state. This drives the entire downstream pipeline.
5
LangGraph-Inspired ChatGraph Dispatch
A hand-rolled state graph inspired by LangGraph's explicit-state, conditional-routing, trace-logging pattern — implemented without the LangChain dependency overhead.

Model Allocation Strategy

With 32 GB unified memory shared between macOS, the GPU, and all running processes, I can't keep every model loaded. The keep_alive parameter in Ollama is the key lever:

Qwen 3 8B ~5 GB Quality Generation 2–3s 5 min Best quality-per-GB for bilingual (ZH/EN). Strong instruction following, good at structured JSON extraction
Tifa-Roleplay 14B Q4 ~8 GB Character / Roleplay 3–5s 30s Specialized fine-tune for character consistency. Short keep_alive to free memory when not in character mode

💡
Key tradeoff: The 14B character model can't coexist with the 8B quality model in memory — loading one evicts the other. I use a 30-second keep_alive on the 14B to aggressively unload it when not actively in character mode, freeing memory for the quality model 95% of the time.

RAG Memory System

Long-term memory uses pgvector for cosine similarity search over embedded conversation summaries:

Session Ends → LLM Extract (Qwen 8B) → Embed (nomic-embed-text) → INSERT pgvector
New Message → Embed Query → Cosine Search (top-5) → Inject into System Prompt

A dual-layer cleanup system (rule-based + normalized dedup) prevents low-quality memories from polluting retrieval. Importantly, RAG cleanup is a manual user-triggered action via a dashboard button—not automatic—because the user should control when history is forgotten.

✅
Graceful degradation: If PostgreSQL or Ollama is unavailable, the system automatically falls back to in-memory recency-based search. The app never crashes — it just loses multi-session context until services recover.

Sliding Window Context & Session Compression

Small local models have limited context windows. Without management, after ~14 turns the model starts losing track of earlier conversation topics. My solution uses a sliding window with progressive summarization:

  • After 14 messages, a compressor node summarizes older turns into a running session["summary"]
  • Only the 6 most recent messages are kept verbatim — the summary carries forward everything else
  • The summary preserves entities (people, companies, topics) and open questions so the model can resolve follow-up references ("it", "those two", "the company mentioned earlier")
  • This keeps the effective context focused for the small 8B model while maintaining conversational coherence across an entire session

The key insight: rather than blindly truncating old messages (which breaks multi-turn reasoning), progressive summarization gives the model a compressed but semantically complete view of the full conversation.

Anti-Repetition & Response Quality Control

Character-mode LLMs love to repeat themselves — generating the same compliments, same actions, same ending questions. I built a multi-layer defense:

  1. Jaccard n-gram similarity — After generation, the response is compared against the previous assistant turn using 4-gram character sets. If Jaccard similarity exceeds the threshold (configurable, default 0.4), the response is flagged as repetitive.
  2. Substring containment check — Catches cases where the model copies entire clauses verbatim even if the overall n-gram score is below threshold
  3. Rule validator — Detects repetitive phrases within a single response (clause-level dedup), vague responses to decision/comparison tasks, and empty outputs
  4. LLM validator (Qwen 8B) — Scores response relevance, context usage, and goal alignment via structured JSON verdict. Failed responses are auto-repaired by re-prompting with explicit guidance ("avoid repeating..." + original response examples).

2 3D Avatar & Animation

LLM-Driven Interactive VRM Avatar

Having taken a Video Game Design and Computer Graphics course where I worked with Unity and animation pipelines, I wanted to bring the chat companion to life with a real-time 3D avatar whose behavior is driven entirely by LLM output.

Tech Stack

Three.js r161 @pixiv/three-vrm 2.0 @pixiv/three-vrm-animation 3.5 Mixamo FBX VRM 1.0 Format Web Audio API

LLM-Driven Animation Selection

The key innovation: the LLM decides which animation to play, not the frontend. During the Turn Understanding phase, the model outputs structured fields for avatar_action and avatar_mood:

User: "Can you dance?" → Turn Understanding → avatar_action: "dance" → Play Swing Dancing.fbx
User: "I feel sad today" → Turn Understanding → avatar_mood: "sad" → Consoling pose + response

This means the avatar reacts semantically to conversation content — it dances when asked, looks sad when the user is upset, waves on greeting — all without hardcoded triggers.

Avatar State Machine

The avatar responds to 4 conversation states (idle, listening, thinking, speaking) and 5 mood states (neutral, happy, sad, angry, thankful). Each state maps to blend tree transitions in Three.js for smooth interpolation between clips.

Supported actions: dance, wave, salute, point, thankful, excited, sad, angry — each mapped to a Mixamo FBX clip that's retargeted to the VRM skeleton at runtime. Skill-specific voice overs trigger matching animations.

Lip Sync via Web Audio API

Real-time lip sync is driven by audio analysis, not phoneme mapping. When TTS audio plays: MediaElementSource → AnalyserNode (FFT 256, smoothing 0.82) → RMS volume → morph target blend weights. The mouth opens wider proportionally to loudness, creating believable talking animation without speech recognition.

🎮
From Unity to Three.js: My background in Unity gave me the vocabulary (blend trees, animation controllers, IK rigs). The web approach trades some control for zero-install deployment and browser-native compatibility.

Streaming Orpheus: Token Budget & Buffer Optimization

The most challenging part was making Orpheus work in real-time streaming. Orpheus is an LLM that generates audio as <custom_token_N> tokens decoded by the SNAC codec. Here's the engineering:

Bilingual Model Switching

The system runs two separate Orpheus models — an English base model and a Chinese fine-tuned variant (orpheus-3b-chinese-ft). Language detection happens inline by counting Chinese characters:

Token Budget Estimation

Orpheus tokens don't map 1:1 to text characters. Through empirical testing across ~50 utterances, I derived the formula:

// Token budget: Chinese chars need ~35 tokens each, English chars ~12 max_tokens = max(600, min(1200, zh_chars * 35 + (len(text) - zh_chars) * 12))

Too few tokens = audio cuts off mid-sentence. Too many = wasted compute and potential repetition loops. The formula was calibrated by generating ~50 test utterances and measuring token consumption empirically.

SNAC Decode Buffer Strategy

SNAC uses a hierarchical 3-codebook structure: every 7 tokens form one audio frame. The streaming pipeline must accumulate tokens before decoding:

// Only decode after accumulating enough frames (28 tokens = 4 frames) if count % 7 == 0 and count > 27: chunk = _convert_tokens_to_audio(buffer[-28:]) // 4-frame lookback window

Why 28-token lookback? SNAC's decoder produces better quality when given multi-frame context rather than single frames. Testing showed 4-frame windows (28 tokens) balanced latency (~250ms per chunk) with audio fidelity.

Repetition Detection in Audio Stream

LLM-based TTS models sometimes enter repetition loops, generating the same audio pattern indefinitely. I implemented a 3-chunk ring buffer detector:

// If last 3 audio chunks are identical bytes, the model is looping last_3.append(chunk) if len(last_3) > 3: last_3.pop(0) if len(last_3) == 3 and last_3[0] == last_3[1] == last_3[2]: break // early exit, save compute

SSE Streaming Architecture

LLM Text → LM Studio Stream → SNAC Decode → SSE audio chunks → Browser PCM

The streaming endpoint sends three event types: event:meta (text + metadata), event:audio (base64 PCM chunks, ~85ms each at 24kHz), and event:done. The producer maintains a ring buffer of audio frames to handle backpressure if the browser lags.

Sentence Segmentation for Quality

Long text degrades TTS quality significantly. My 5-pass segmenter splits text into optimal 15–80 character segments:

  1. Split on sentence endings (periods, exclamation marks, question marks, ellipsis)
  2. If only 1 segment and >60 chars, split on commas and semicolons
  3. Merge short fragments (<15 chars) into neighbors — prevents choppy audio
  4. Re-split any segment >80 chars on comma boundaries
  5. Merge trailing orphan (<12 chars) back into previous — avoids isolated particles
⚠️
Hardware constraint insight: On M1 Max, the Orpheus 3B model (running via LM Studio) shares GPU memory with the chat LLMs (via Ollama) and Draw Things. I can't run all three simultaneously at full capacity—Orpheus typically gets ~6GB, Ollama ~8GB, and Draw Things ~4GB, with 14GB reserved for macOS.
4 Voice Input & Speech-to-Text

Local Whisper-Based STT

Voice input uses faster-whisper running locally on CPU with INT8 quantization:

Model Size Latency (M1) Chinese Accuracy Notes
tiny 39 MB ~0.3s ✗ Too many errors Homophone confusion renders Chinese unusable
base ★ 74 MB ~0.5s ✓ Good Default. Sweet spot for casual bilingual use
small 244 MB ~1s ✓ Much better Recommended for production Chinese
medium 769 MB ~2.5s ✓ Near-perfect Best if latency is acceptable

Key optimizations for Chinese accuracy:

  • Initial prompt injection — A Chinese conversational prompt steers Whisper away from misreading common words as literary terms or place names
  • Two-pass detection — First pass detects language (beam_size=1, fast), second pass transcribes with language-specific prompt (beam_size=5, best_of=5)
  • VAD filtering — min_silence_duration=300ms to avoid splitting mid-word
  • condition_on_previous_text=False — Prevents hallucination loops where Whisper repeats itself
5 Image Generation

Stable Diffusion via Draw Things

Image generation runs through Draw Things on macOS, which provides an SD WebUI-compatible HTTP API at localhost:7860. I built a full-featured web UI and tested three different SD variants:

Model Comparison

Model Steps CFG Sampler Time (M1 Max) Best For
RealVisXL V5.0 30 4.5 DPM++ 2M Karras ~45s Photorealistic portraits, landscapes. Best overall quality with realistic lighting
PonyDiffusion V6 XL 35 7.0 Euler A ~55s Anime/illustration style. Requires score_9 quality tags, clip_skip=2. Largest LoRA ecosystem
Z Image Turbo 1.0 8 1.5 Euler A Trailing ~15s Speed-optimized. 4x faster than RealVisXL. Uses zero_negative_prompt. Good for rapid iteration
💡
Draw Things constraint: Only one model is loaded at a time — there's no API for model switching. Width and height must be multiples of 64 and must match the init_image dimensions for img2img workflows. This forces strategic model selection upfront.

Parameter Deep-Dive

  • CFG Scale (Guidance) — How closely the model follows the prompt. Low (1.5 for Z Image) = creative freedom, medium (4.5 for RealVis) = balanced, high (7.0 for Pony) = strict adherence but can look artificial
  • Steps — More steps = more detail but diminishing returns past ~30. Z Image Turbo is specifically trained to converge at 8 steps
  • Sampler — DPM++ 2M Karras gives the most consistent photorealistic results. Euler A is faster but noisier. Euler A Trailing is optimized for turbo models
  • clip_skip — PonyDiffusion requires clip_skip=2 (skipping the last CLIP layer) because it was trained that way. Using clip_skip=1 produces washed-out images
  • Negative Prompt — Z Image Turbo uses zero_negative_prompt=true — its training explicitly teaches the model what not to generate without needing a negative prompt string
